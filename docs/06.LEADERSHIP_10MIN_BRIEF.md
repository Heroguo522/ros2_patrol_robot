## 3. 我做了什么

### 3.1 从“巡逻 demo”升级为“端侧任务架构原型”

原项目是：

```text
导航到点位 -> 语音播报 -> 拍照 -> 下一个点
```

我扩展后变成：

```text
云端/MQTT 下发任务
        |
        v
TaskManager 管理任务生命周期
        |
        v
TaskOrchestrator 按 YAML DSL 编排步骤
        |
        v
原子 Skill 执行导航、拍照、检测、语音、打螺丝、上报
        |
        v
FaultManager 处理异常，Gateway 上报状态和事件
```

### 3.2 当前工程新增模块

| 模块 | 作用 |
|---|---|
| `TaskManager` | 任务生命周期、状态管理、远程任务接入 |
| `TaskOrchestrator` | 按 YAML 步骤顺序执行任务 |
| `SkillRegistry` | 注册和调用原子 Skill |
| `skills/*` | 导航、拍照、检测、语音、打螺丝、上报 |
| `robot_gateway_node` | MQTT 遥测、事件、命令、ACK |
| `FaultManager` | 故障分类、恢复、上报 |
| `config/tasks/*.yaml` | YAML 任务 DSL |
| `industrial_screw_inspection.yaml` | 工业复合任务 Demo |

## 5. Demo：工业复合任务

Demo 任务文件：

```text
patrol_robot/config/tasks/industrial_screw_inspection.yaml
```

任务流程：

```text
1. 导航到 1 号工位
2. 语音播报：到达工位，开始检测和打螺丝
3. 拍照保存工件图片
4. Mock 工件检测：判断是否 ready_for_screw
5. Mock 机械臂 + 电批执行打螺丝
6. 上报工位结果
7. 导航到 2 号工位，重复以上流程
8. 返回 home
9. 上报任务总结
```

MQTT 下发命令：

```bash
mosquitto_pub -h 127.0.0.1 -t 'robots/robot_001/command' -m \
  '{"command_id":"demo1","action":"start_task","task_name":"industrial_screw_inspection","task_id":"industrial_screw_demo"}'
```

MQTT 订阅观察：

```bash
mosquitto_sub -h 127.0.0.1 -t 'robots/robot_001/#' -v
```

可展示的输出：

- `telemetry`：机器人状态、任务进度、位姿、电量、故障码；
- `events`：命令接收、状态变化、任务报告、故障事件；
- `/robot/composite_task_report`：工位级复合任务结果；
- 图片保存目录：工件拍照记录；
- 日志：Mock 机械臂和 Mock 电批执行过程。

## 6. 故障管理能力

当前原型不是只做“正常流程”，还加入了工业机器人端侧必需的故障闭环：

| 故障类型 | 示例故障码 | 默认策略 |
|---|---|---|
| 导航故障 | `NAV_FAILED`, `NAV_TIMEOUT`, `TF_UNAVAILABLE` | 重试、清 costmap、失败上报 |
| 相机故障 | `CAMERA_NO_IMAGE`, `CAMERA_STALE_IMAGE` | 默认告警并继续，可配置为阻断 |
| 语音故障 | `TTS_TIMEOUT`, `TTS_FAILED` | 默认不阻断任务 |
| 上报故障 | `REPORT_FAILED` | 告警并继续 |
| 编排异常 | `STEP_EXCEPTION`, `TASK_FAILED` | 中止任务并上报 |

故障生命周期：

```text
fault_detected -> recovery_started -> recovery_retrying -> recovery_succeeded / recovery_failed -> task_failed
```

这部分体现的是工业机器人端侧系统的可靠性设计，而不是单纯功能演示。
